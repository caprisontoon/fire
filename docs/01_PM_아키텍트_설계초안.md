# [1차 산출물] 총괄 PM/아키텍트 — 요구사항·우선순위·아키텍처 초안

> 작성 역할: 총괄 PM / 아키텍트 에이전트
> 상태: **초안 (사용자 확인 대기)** — "OK" 확인 후 2단계(세무·건보 규칙 상세 설계)로 진행
> 대상 독자: 비개발자인 프로젝트 오너 + 후속 작업할 각 역할 에이전트

---

## 1) 요약 (3분 안에 읽는 핵심)

- **무엇을 만드나:** 한국 은퇴 준비자·파이어족이 자신의 소득·자산을 **전부 수동 입력**하면,
  ① 금융소득종합과세 대상 여부, ② 예상 종합소득세, ③ 예상 건강보험료와 피부양자 유지 여부,
  ④ 90세까지의 연간 순 현금흐름, ⑤ 시나리오 비교(금융소득 1,000만/2,000만 관리)를 보여주는 **웹 시뮬레이터**.
- **어떻게 만드나:** 회원가입·DB 없이, **브라우저에서 입력 → 서버(또는 브라우저 내) 계산 → 결과 표시**의
  단순한 1회성 계산 구조. 개인 금융정보를 서버에 저장하지 않는 것이 원칙.
- **기술 스택 추천:** 백엔드 **Python + FastAPI**(계산 엔진), 프론트엔드 **단일 페이지 HTML + JavaScript**(빌드 도구 없음).
  비개발자가 로컬에서 명령어 2~3개로 띄울 수 있는 가장 단순한 조합.
- **진행 순서:** ① 본 설계 확인 → ② 세무·건보 규칙 설계 → ③ 데이터 모델/API 스펙 확정 →
  ④ 백엔드 계산 엔진 구현 → ⑤ 프론트엔드 구현 → ⑥ 시나리오/현금흐름 로직 → ⑦ 문구·면책 적용 → ⑧ 통합 테스트.

## 2) 요구사항 정리: MVP vs 차후 확장

### 2-1. MVP (1차 출시 버전에 반드시 포함)

| # | 기능 | 설명 |
|---|------|------|
| M1 | 수동 입력 6단계 폼 | 기본 정보 → 근로/사업/기타소득 → 금융소득 → 연금 → 재산/임대 → 결과 |
| M2 | 금융소득 합산 및 종합과세 판정 | 이자+배당+ETF 분배금+보유기간과세 매매차익 합산, 2,000만 원 초과 여부 |
| M3 | 종합소득세 단순 추정 | 단순화된 누진세율 모델. 사적연금 분리과세/종합과세 단순 반영 |
| M4 | 건강보험료 추정 | 직장/지역/피부양자 3가지 자격별 단순 모델 |
| M5 | 피부양자 유지 여부 판정 | 합산소득 2,000만 원 + 재산 과표 기준의 단순화 판정 |
| M6 | 연령별 순 현금흐름 시뮬레이션 | 현재 나이~90세, 연 단위. 세후·건보료 차감 후 순 현금흐름 |
| M7 | 자산 소진 시점 계산 | 단순 수익률 가정 + 4% 룰 참고, "자산이 몇 살까지 버티는가" |
| M8 | 시나리오 비교 (A/B/C) | A: 현재 입력 그대로 / B: 금융소득 2,000만 이하 관리 / C: 1,000만 이하 관리 |
| M9 | 면책·주의 문구 | "참고용 시뮬레이터" 면책, 각 필드 안내 문구 |
| M10 | 결과 화면 | 카드(판정 결과) + 표/그래프(연령별 현금흐름) |

### 2-2. 차후 확장 (MVP 이후)

| # | 기능 | 비고 |
|---|------|------|
| F1 | 사적연금 분리과세 vs 종합과세 선택 시나리오 | 시나리오 D로 추가 |
| F2 | 국가법령정보센터 Open API 연동 | 관련 법령 링크·설명 표시용 (필수 아님) |
| F3 | 입력값 저장/불러오기 | 브라우저 localStorage 또는 JSON 파일 내보내기 (서버 저장 없음) |
| F4 | 물가상승률·연금 인상률 등 거시 가정 커스터마이징 | MVP에서는 고정 가정값 사용 |
| F5 | 부부 합산 시뮬레이션 | MVP는 1인 기준 (배우자는 피부양자 판정 참고용만) |
| F6 | 마이데이터/홈택스 연동 | 명시적으로 MVP 제외. 아이디어로만 보관 |
| F7 | 건보료 정밀 모델 (재산점수표, 자동차 등) | MVP는 단순 부과율 모델 |
| F8 | ISA·연금계좌 내 운용 전략 비교 | 절세 전략 고도화 |

### 2-3. 명시적 제외 (만들지 않음)

- 회원가입·로그인·개인정보 서버 저장
- 실제 세무 신고 서식 출력 (홈택스 신고 대체 아님)
- 마이데이터·홈택스 B2B 연동

## 4) 아키텍처 제안

### 4-1. 기술 스택 결정 (추천안)

| 영역 | 선택 | 이유 |
|------|------|------|
| 백엔드 | **Python 3.11+ + FastAPI** | 세금 계산 같은 수식 로직을 읽기 쉬운 코드로 작성 가능. Pydantic으로 입력 검증 자동화. `uvicorn main:app` 한 줄로 실행. AI 코딩 도구와 궁합이 가장 좋음 |
| 프론트엔드 | **단일 HTML + Vanilla JS + Chart.js(CDN)** | npm/빌드 과정이 전혀 필요 없음. 파일 하나 열면 동작. 비개발자 운영에 최적 |
| 정적 파일 서빙 | FastAPI가 프론트 HTML도 함께 서빙 | 서버 1개만 띄우면 끝 (포트 1개) |
| 데이터베이스 | **없음** | 개인정보 비저장 원칙. 모든 계산은 요청 1회로 완결 |
| 그래프 | Chart.js (CDN) | 스크립트 태그 한 줄, 연령별 라인 차트에 충분 |
| 배포(차후) | 로컬 실행 → 필요 시 Render/Railway 등 무료 호스팅 | MVP는 로컬 실행으로 충분 |

> **대안 검토:** React는 결과물 품질은 좋지만 Node/npm/빌드 학습 부담이 있어 비개발자 단독 운영에는 과합니다.
> Node.js+Express도 가능하지만, 세율표·수식 중심 코드는 Python이 더 읽기 쉽습니다. → FastAPI 추천.

```
fire/
├── README.md                  # 설치·실행 방법 (비개발자용 한국어 안내)
├── docs/                      # 모든 설계 문서 (역할별 산출물)

│   ├── 03_데이터모델_API스펙.md        ← 백엔드 설계 산출물
│   ├── 04_화면설계.md                  ← 프론트엔드 설계 산출물
│   ├── 05_시뮬레이션_시나리오설계.md   ← 데이터/시뮬레이션 산출물
│   └── 06_UX_카피.md                   ← UX·카피 산출물
├── backend/
│   ├── main.py                # FastAPI 앱 + /api/simulate 엔드포인트
│   ├── models.py              # 입력/출력 데이터 모델 (Pydantic)
│   ├── rules.py               # 세율표·기준값 상수 (세법 개정 시 이 파일만 수정)
│   ├── tax.py                 # 금융소득 합산·종합과세 판정·소득세 계산
│   ├── health_insurance.py    # 건보료·피부양자 판정
│   ├── simulation.py          # 연령별 현금흐름·시나리오 실행
│   └── tests/                 # 계산 검증 테스트 (대표 케이스)
├── frontend/
│   ├── index.html             # 6단계 입력 + 결과 화면 (단일 페이지)
│   ├── app.js                 # 폼 처리, API 호출, 결과 렌더링
│   └── style.css              # 스타일
└── requirements.txt           # Python 의존 패키지 (fastapi, uvicorn)
```

**설계 의도:** 세법·건보 기준값(세율, 2,000만 원 기준선, 부과율 등)은 전부 `rules.py` 한 파일에 상수로 모아,
제도가 바뀌면 **숫자만 바꾸면 되도록** 합니다. 계산 로직과 기준값을 분리하는 것이 이 프로젝트의 핵심 설계 원칙입니다.

### 4-3. API 스펙 초안 (단일 엔드포인트)

```
POST /api/simulate
```

```json

  "basic": { "current_age": 52, "retire_age": 55, "married": true,
             "insurance_status": "직장" },
  "income": { "earned": 60000000, "business": 0, "other": 0 },
  "financial": {
    "interest": 3000000,
    "dividend_domestic": 5000000,
    "dividend_foreign": 4000000,
    "etf_domestic_equity_dist": 1000000,
    "etf_domestic_other_dist_and_gain": 6000000,
    "etf_foreign_dist": 2000000,
    "capital_gain_foreign": 10000000
  },
  "pension": { "public_annual": 18000000, "private_annual": 12000000,
               "tax_free_annual": 0, "housing_monthly": 0 },
  "property": { "real_estate_tax_base": 300000000,
                "rent_monthly": 500000, "financial_assets": 800000000 }
}
```

**응답(JSON) 골격:**

```json
{
  "aggregate_financial_tax": { "subject": true, "total": 21000000,
                               "threshold": 20000000, "message": "..." },
  "income_tax": { "estimated_annual": 4500000, "details": "..." },
  "health_insurance": { "monthly_premium": 280000, "status": "지역",
                        "dependent_eligible": false, "message": "..." },
  "cashflow_by_age": [
    { "age": 55, "income": 35000000, "tax": 3000000,

  "scenarios": { "A": { "...": "위와 동일 구조" }, "B": {}, "C": {} },
  "disclaimers": ["이 결과는 단순화된 모델로..."]
}
```

### 4-4. 데이터 흐름

```
[브라우저: 6단계 입력 폼]
        │  (입력 완료 → JSON 1건)
        ▼
POST /api/simulate  ──►  검증(Pydantic) ──► 금융소득 합산 ──► 종합과세 판정
                                              │
                          소득세 추정 ◄───────┤
                          건보료/피부양자 ◄───┤

        ▼

```

## 6) 핵심 설계 결정과 그 이유 (오너가 알아야 할 것)

1. **DB 없음 / 저장 없음** — 민감한 개인 금융정보를 다루므로, 서버에 아무것도 남기지 않는 구조가
   가장 안전하고 구현도 단순합니다. 저장이 필요하면 차후 브라우저 localStorage(F3)로 해결합니다.
2. **기준값과 로직의 분리 (`rules.py`)** — 세법·건보 기준은 매년 바뀝니다. 숫자만 바꾸면 되는 구조로
   유지보수 부담을 최소화합니다.
3. **해외 주식/ETF 양도차익은 금융소득과 별도 트랙** — 종합과세 판정(2,000만 기준)에는 포함하지 않고,
   양도소득세(기본공제 250만, 22%)로 따로 단순 계산합니다. 입력 칸도 별도로 둡니다.
4. **시나리오는 서버에서 한 번에 계산** — A/B/C를 한 요청으로 모두 계산해 내려주면,
   프론트는 받은 것을 그리기만 하면 되어 단순해집니다.
5. **단순화 고지의 의무화** — 모든 판정·금액 옆에 "단순화된 추정"임을 명시하는 것을
   기능 요구사항(M9)으로 격상했습니다. 신뢰성 문제를 문구로 방어합니다.
